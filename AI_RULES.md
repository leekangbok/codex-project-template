# AI 규칙

이 문서는 Codex CLI와 이 프로젝트에서 동작하는 모든 AI 에이전트가 따라야 하는 필수 작업 규칙입니다.
최우선 목표는 속도보다 신뢰 가능한 변경, 최신 문서 유지, 다음 세션으로의 쉬운 인계입니다.

## 절대 규칙

1. 작업 전에 `README.md`, `CONTEXT.md`, `TASKS.md`, `HARNESS.md`를 읽습니다.
2. 프로젝트 핵심 정보가 비어 있으면 먼저 사용자에게 필요한 항목을 질문하거나 `project-intake`를 실행해 초기 정보를 채웁니다.
3. 코드가 바뀌면 관련 문서도 같은 작업 안에서 수정합니다.
4. 코드 수정 또는 변경 후에는 관련 테스트나 검증을 먼저 수행하고, 통과 전에는 다음 단계로 진행하지 않습니다.
5. 가정은 명확히 표시하고, 추정을 사실처럼 말하지 않습니다.
6. 다음 세션이 재탐색 없이 바로 이어갈 수 있도록 문맥을 남깁니다.

## 문서 강제 업데이트 규칙

아래 사항이 바뀌면 문서 업데이트는 선택이 아니라 필수입니다.

### `README.md`를 업데이트해야 하는 경우

- 프로젝트 목표 또는 범위가 변경됨
- 설치, 실행, 테스트, 린트, 빌드 명령이 변경됨
- 새 기여자를 위한 기본 온보딩 흐름이 변경됨

### `docs/architecture.md`를 업데이트해야 하는 경우

- 디렉터리 구조가 변경됨
- 모듈 경계 또는 책임이 변경됨
- 외부 의존성 또는 데이터 흐름이 변경됨
- 중요한 기술 의사결정이 이루어짐

### `TASKS.md`를 업데이트해야 하는 경우

- 새로운 작업이 생김
- 우선순위가 바뀜
- 작업이 완료, 보류, 취소, 연기됨

### `CONTEXT.md`를 업데이트해야 하는 경우

- 현재 프로젝트 상태가 바뀜
- 다음 권장 작업이 바뀜
- 알려진 리스크나 막힘이 바뀜
- 세션 종료 전 인수인계가 필요함

## 필수 작업 흐름

1. 변경 전 현재 문서를 읽습니다.
2. 코드 변경을 수행합니다.
3. 가능하면 `scripts/check.*`를 통해 관련 테스트 또는 검증을 실행합니다.
4. 테스트가 통과한 뒤에만 다음 단계로 진행합니다.
5. 영향받는 모든 문서를 같은 작업에서 업데이트합니다.
6. 구조나 상태가 바뀌면 `python scripts/auto_codex_context.py`를 실행합니다.
7. 중단하기 전에 `TASKS.md`, `CONTEXT.md`가 실제 상태와 일치하는지 확인합니다.
8. 검증을 하지 못했다면 그 사실을 명시합니다.

## 구현 규칙

- 작고 검토하기 쉬운 변경을 선호합니다.
- 기존 프로젝트 구조와 스타일을 존중합니다.
- 요청된 작업에 꼭 필요한 경우가 아니면 큰 리팩터링은 피합니다.
- 테스트가 있다면 관련 테스트를 추가하거나 갱신합니다.
- 테스트 가능한 변경은 반드시 테스트로 검증한 뒤 다음 단계로 넘어갑니다.
- 자동 생성 영역과 수동 관리 영역은 명확히 구분합니다.

## 기록 규칙

- 설계 결정은 `docs/architecture.md`에 기록합니다.
- 다음 작업은 `TASKS.md`에 기록합니다.
- 현재 상태와 블로커는 `CONTEXT.md`에 기록합니다.
- 사용법이나 작업 흐름 변경은 `README.md`에 기록합니다.
- 검증 전략이나 표준 체크 명령 변경은 `HARNESS.md`에 기록합니다.

## 금지 사항

- 코드와 문서가 어긋난 상태로 종료
- 테스트 통과 전 다음 단계 진행
- 검증 상태 확인 없이 "완료"라고 선언
- 근거 없이 설계 결정을 확정
- 요청받지 않은 큰 범위 변경 수행
- `CONTEXT.md`의 자동 생성 영역 형식을 깨뜨림

## 작업 종료 체크리스트

- [ ] 코드와 문서가 동기화되어 있는가
- [ ] 코드 변경 후 테스트 또는 검증이 통과했는가
- [ ] 관련 작업 목록이 업데이트되었는가
- [ ] 현재 문맥이 업데이트되었는가
- [ ] 검증 상태가 명확한가
- [ ] 남은 리스크가 기록되었는가
